## 2. プロジェクト概要

### 2.1 目的

「読む・聞く」中心になりがちな英語学習を、**書く（英作文）・話す（発音）のアウトプット中心**の習慣に変える。毎日の日記を題材に、自力での英訳 → 語彙調査 → 再英訳 → AI 添削 → シャドーイング → 発音チェックという一連のサイクルを 1 つのアプリで完結させる。

### 2.2 背景

- 単なる翻訳アプリでは「自分で考えて書く」プロセスが省略され、学習効果が薄い
- 添削・発音練習・単語帳・テストが別々のツールに分散していると継続しにくい
- 無料枠の外部サービス（Gemini API / Supabase / GitHub Pages / Web Speech API）を組み合わせることで、**運用費 0 円**で実現できる

### 2.3 ターゲットユーザー

- 学習者本人による**個人利用**（シングルユーザー前提）
- 想定レベル：日本語話者の英語学習者（初級〜中級）
- 利用端末：Mac / iPhone などのブラウザ（iPhone は「ホーム画面に追加」で PWA ライクに利用）

### 2.4 スコープ

- 対象：日記作成の学習フロー、AI 添削、音声再生・発音チェック、単語帳、テスト、UI 言語切り替え
- 対象外：ユーザー認証、複数ユーザーのデータ分離、オフライン動作、ネイティブアプリ化

## 3. システム構成

### 3.1 構成概要

サーバーサイドのコードを持たない**静的シングルページアプリケーション（SPA）**。ブラウザから外部サービスの API を直接呼び出す。

```
┌────────────────────────────────────────────┐
│  ブラウザ（GitHub Pages で配信される静的 SPA）│
│  index.html / style.css / app.js           │
│  ├─ localStorage（API キー・UI 言語設定）    │
│  ├─ Web Speech API（TTS / 音声認識）※内蔵   │
│  │                                         │
│  ├──→ Gemini API（gemini-2.5-flash）        │
│  │      英文添削・フィードバック生成          │
│  │                                         │
│  └──→ Supabase（PostgreSQL）                │
│         entries（日記）/ vocab（単語帳）      │
└────────────────────────────────────────────┘
```

### 3.2 技術スタック

| 区分 | 技術 | 用途 |
|------|------|------|
| フロントエンド | HTML / CSS / Vanilla JavaScript | 画面・全ロジック（ビルド工程なし） |
| ホスティング | GitHub Pages | 静的ファイル配信 |
| データベース | Supabase（PostgreSQL, supabase-js v2 CDN） | 日記・単語帳の永続化 |
| AI 添削 | Google Gemini API（`gemini-2.5-flash`） | 英文添削・フィードバック |
| 音声合成（TTS） | Web Speech API `speechSynthesis` | 添削後英文の読み上げ |
| 音声認識 | Web Speech API `SpeechRecognition` | 発音チェック |
| フォント | Google Fonts（Inter / Noto Sans JP） | UI 表示 |

### 3.3 画面構成

| 画面 | 内容 |
|------|------|
| セットアップ画面 | 初回アクセス時に API キー 3 種を入力（Gemini / Supabase URL / Supabase anon key） |
| 日記タブ | 6 ステップの学習フローカード ＋ 過去の日記一覧 |
| 単語帳タブ | 単語の追加・一覧・削除・正答率表示 |
| テストタブ | 単語帳を使ったクイズ |
| 日記詳細モーダル | 過去日記の日本語・英訳 1 回目・2 回目・AI 添削文の表示と音声再生 |
| 設定モーダル | API キーの変更 |

## 4. 機能要件（実装済み）

### F-01 初期セットアップ・設定

| ID | 要件 |
|----|------|
| F-01-1 | 初回アクセス時（キー未設定時）、セットアップ画面を表示し、Gemini API Key / Supabase URL / Supabase Anon Key の 3 項目を必須入力とする |
| F-01-2 | 入力されたキーはブラウザの localStorage にのみ保存する（サーバー送信しない） |
| F-01-3 | 設定モーダル（⚙️）からいつでもキーを変更できる |

### F-02 日記作成 6 ステップフロー

各ステップはカード UI で表示され、前のステップを完了するまで次のカードはロック状態とする。

| ID | ステップ | 要件 |
|----|---------|------|
| F-02-1 | ① 日本語で日記を書く | 当日日付を表示し、日本語で自由記述。未入力時は次へ進めない |
| F-02-2 | ② 自力英訳（1 回目） | ①の日本語を参照表示した状態で、辞書等を見ずに英訳を記入 |
| F-02-3 | ③ わからない単語をメモ | 「日本語 → 英語」の単語ペアを任意個数追加・削除できる。ここで単語を調べる |
| F-02-4 | ④ 再英訳（2 回目） | ①の日本語と③で調べた単語をヒント表示した状態で、改善版の英訳を記入 |
| F-02-5 | ⑤ AI 添削 | ④の英文を Gemini API に送信し、添削結果・フィードバック・修正ポイントを表示（詳細は F-03） |
| F-02-6 | ⑥ シャドーイング・発音チェック | 添削後英文を TTS で再生（F-04）し、音声入力で発音を採点（F-05）した後、日記を保存（F-07） |

### F-03 AI 添削（Gemini API）

| ID | 要件 |
|----|------|
| F-03-1 | 日本語原文・英訳 1 回目・調べた単語リスト・英訳 2 回目をプロンプトに含めて `gemini-2.5-flash` に送信する |
| F-03-2 | レスポンスは JSON スキーマ指定（`responseMimeType: application/json` + `responseSchema`）で構造化して受け取る。フィールド：`corrected`（添削後英文）/ `good_points`（良かった点 2 件）/ `improvements`（改善点 1〜2 件）/ `corrections`（重要な修正 3〜6 件。before / after / type / explanation）/ `vocab_usage`（調べた単語の使い方への講評） |
| F-03-3 | 修正タイプは grammar / vocabulary / naturalness / spelling の 4 分類とし、色付きタグで表示する |
| F-03-4 | フィードバック文の言語は UI 言語設定（日/英）に追従する |
| F-03-5 | API が 429 または 5xx を返した場合、2.5 秒待って 1 回だけ再試行する |
| F-03-6 | 添削に失敗した場合は、学習者の英文（2 回目）をそのまま添削後英文として扱い、エラーメッセージを併記してフローを継続できるようにする |
| F-03-7 | 生成温度は 0.3 とし、出力のばらつきを抑える |

### F-04 音声再生（TTS・シャドーイング）

| ID | 要件 |
|----|------|
| F-04-1 | 添削後英文をブラウザ内蔵の `speechSynthesis` で読み上げる（言語 en-US、速度 0.9 倍、英語ボイスを自動選択） |
| F-04-2 | ステップ⑥に進んだ際、目標文を自動で 1 回読み上げる |
| F-04-3 | 何度でも再生でき、過去日記の詳細モーダルからも再生できる |

### F-05 発音チェック（音声認識）

| ID | 要件 |
|----|------|
| F-05-1 | `SpeechRecognition`（en-US、候補 3 件）で学習者の発話を認識する。非対応ブラウザでは警告を表示する |
| F-05-2 | 認識テキストと目標文を正規化（小文字化・アクセント記号除去・短縮形展開・記号除去）した上で、LCS（最長共通部分列）により語順を考慮して単語単位で照合する |
| F-05-3 | 一致率（%）をスコアとして表示する。80% 以上 / 50% 以上 / 50% 未満の 3 段階で評価メッセージを変える |
| F-05-4 | 目標文を単語ごとに色分け表示する（一致＝緑、不一致＝赤）。認識候補 3 件のうち最高スコアを採用する |

### F-06 過去日記の閲覧

| ID | 要件 |
|----|------|
| F-06-1 | 保存済み日記を日付降順で最大 50 件一覧表示する（日付＋日本語冒頭プレビュー） |
| F-06-2 | 一覧のカードをタップすると詳細モーダルを表示する（日本語 / 英訳 1 回目 / 2 回目 / AI 添削後、音声再生ボタン付き） |

### F-07 日記の保存

| ID | 要件 |
|----|------|
| F-07-1 | 日付・日本語・英訳 1 回目・2 回目・添削後英文を `entries` テーブルに保存する。日本語未入力時は保存不可 |
| F-07-2 | ステップ③でメモした単語（日英とも入力済みのもの）を `vocab` テーブルへ自動追加する。英語表記の大文字小文字を無視して重複を除外する |
| F-07-3 | 保存後は全ステップを初期状態に戻し、追加された単語数を含む完了トーストを表示する |

### F-08 単語帳

| ID | 要件 |
|----|------|
| F-08-1 | 英語・日本語（必須）とメモ（任意）を手動で追加できる |
| F-08-2 | 登録単語を新しい順に一覧表示し、単語数を表示する |
| F-08-3 | 各単語にテストの累計正答率を表示する（未テスト / 70% 以上＝良好 / 70% 未満＝要復習 の 3 区分で色分け） |
| F-08-4 | 単語を削除できる（確認ダイアログあり） |

### F-09 テスト（単語クイズ）

| ID | 要件 |
|----|------|
| F-09-1 | 単語帳の全単語を出題対象とし、単語ごとに日→英 / 英→日をランダムに決定する |
| F-09-2 | 間違い回数に応じた重み（weight = 1 + wrong × 2）で苦手な単語を序盤に優先出題する（後半はシャッフル） |
| F-09-3 | 解答判定は次の正規化・許容ルールに従う：trim・小文字化・NFKC 正規化・カタカナ→ひらがな変換、複数訳語（、 , / ・ ; 区切り）のいずれか一致で正解、英語は冠詞（a / an / the）を無視、5 文字以上の英語は編集距離 1 のタイポを正解扱い（タイポ注記付き） |
| F-09-4 | 正解 / 不正解を `vocab` の correct / wrong に即時反映する |
| F-09-5 | 正解数・不正解数・残り問題数・進捗バーを常時表示し、全問終了時に正答率を表示する |
| F-09-6 | ヒント（メモ欄の内容）の表示・非表示を切り替えられる。スキップした問題は末尾に回す |

### F-10 UI 言語切り替え

| ID | 要件 |
|----|------|
| F-10-1 | ヘッダーのボタンで UI 言語を日本語 / 英語で切り替えられる |
| F-10-2 | 選択言語は localStorage に保存し、次回アクセス時も維持する |
| F-10-3 | 静的テキストは `data-i18n` / `data-i18n-placeholder` 属性と翻訳辞書（TRANSLATIONS）で一括切り替えする。AI フィードバックの言語も追従する（F-03-4） |

## 5. 非機能要件

| 分類 | 要件 |
|------|------|
| 費用 | すべて無料枠で運用する（GitHub Pages 無料 / Supabase 無料 500MB / Gemini API 無料 1 日 1,500 リクエスト / Web Speech API はブラウザ内蔵） |
| 性能 | Gemini API 無料枠のレート制限（15 リクエスト/分）内で動作する。レート超過時は自動リトライ（F-03-5） |
| 対応ブラウザ | Web Speech API（TTS・音声認識）に依存するため Chrome / Safari を推奨。音声認識非対応ブラウザでは該当機能のみ警告表示（他機能は利用可） |
| 対応端末 | レスポンシブ対応（viewport 設定済み）。PC・スマートフォンのブラウザで利用可能 |
| 可搬性 | ビルド・サーバー不要の静的ファイル構成。任意の静的ホスティングに配置可能 |
| データ保全 | 学習データはすべて Supabase 上に保存され、複数端末から同一データにアクセスできる |
| XSS 対策 | ユーザー入力・AI 出力を画面表示する際は HTML エスケープを行う |

## 6. データ設計

### 6.1 Supabase テーブル

**entries（日記）**

| カラム | 型 | 説明 |
|--------|----|------|
| id | bigserial PK | 主キー |
| created_at | timestamptz | 作成日時（default now()） |
| date | date NOT NULL | 日記の日付 |
| jp | text NOT NULL | 日本語原文 |
| en1 | text | 自力英訳（1 回目） |
| en2 | text | 再英訳（2 回目） |
| corrected | text | AI 添削後の英文 |

**vocab（単語帳）**

| カラム | 型 | 説明 |
|--------|----|------|
| id | bigserial PK | 主キー |
| created_at | timestamptz | 作成日時（default now()） |
| en | text NOT NULL | 英語 |
| jp | text NOT NULL | 日本語 |
| note | text | メモ（クイズのヒントに使用） |
| correct | int default 0 | テスト正解数（累計） |
| wrong | int default 0 | テスト不正解数（累計。出題重みに使用） |

### 6.2 localStorage キー

| キー | 内容 |
|------|------|
| gemini_key | Gemini API キー |
| sb_url | Supabase プロジェクト URL |
| sb_key | Supabase anon key |
| lang | UI 言語（ja / en。未設定時 ja） |

## 8. セキュリティ・制約事項（既知の課題）

現状は**単一ユーザーの個人利用を前提**とした構成であり、以下の制約を許容している。マルチユーザー化する場合は将来要件（第 9 章）と合わせて解消が必要。

| ID | 内容 |
|----|------|
| C-01 | API キー（Gemini / Supabase anon key）がブラウザの localStorage に平文保存される。共有端末での利用は非推奨 |
| C-02 | Supabase の RLS（Row Level Security）が未設定のため、anon key を知っていれば誰でもデータの読み書きが可能。URL とキーを他者に共有しない運用でカバーしている |
| C-03 | ユーザー認証がなく、データはプロジェクト単位で単一（ユーザーごとの分離なし） |
| C-04 | Gemini API 呼び出しがクライアント直接実行のため、キーがブラウザの開発者ツール等から参照可能 |
| C-05 | 音声認識の精度はブラウザ実装に依存し、発音スコアはあくまで参考値（単語一致率であり、音素レベルの評価ではない） |

## 9. 将来要件（今後の構想）

実装済み機能に対する拡張構想。優先度は 高 → 中 の順。

### 9.1 学習継続機能【優先度：高】

| ID | 要件案 | 補足 |
|----|--------|------|
| R-01 | 連続記録（ストリーク）表示 | entries の date から連続日数を算出し、ヘッダー等に表示。途切れそうな日のリマインドは将来検討 |
| R-02 | 学習カレンダー | 月表示カレンダーに日記を書いた日をマークし、タップで該当日記の詳細を開く |
| R-03 | 統計ダッシュボード | 日記件数・単語帳登録数・テスト正答率の推移をグラフ表示。既存の entries / vocab のデータのみで実現可能 |

### 9.2 復習・出題の強化【優先度：中】

| ID | 要件案 | 補足 |
|----|--------|------|
| R-04 | 間隔反復（SRS）による出題 | vocab に次回出題日・習熟度カラムを追加し、正誤に応じて出題間隔を伸縮（例：SM-2 簡易版）。現行の「間違い回数による重み付け」を置き換え |
| R-05 | 苦手単語の重点復習モード | 正答率が閾値未満（例：70%）の単語のみを対象にしたテストモードを追加 |
| R-06 | 過去日記からの復習出題 | AI 添削の corrections（before → after）を復習問題として再出題し、同じ間違いの反復を防ぐ。entries に corrections の保存カラム追加が必要 |

## 10. 用語集

| 用語 | 説明 |
|------|------|
| シャドーイング | 音声を聞きながら（または直後に）同じ英文を声に出して追いかける練習法 |
| TTS | Text-to-Speech。テキストの音声読み上げ |
| LCS | Longest Common Subsequence（最長共通部分列）。語順を考慮した単語列の一致度算出に使用 |
| NFKC 正規化 | Unicode 正規化形式の一つ。全角/半角などの表記ゆれを統一する |
| 編集距離（Levenshtein 距離） | 2 つの文字列を一致させるのに必要な最小の編集回数。タイポ許容判定に使用 |
| SRS | Spaced Repetition System（間隔反復）。忘却曲線に基づき復習タイミングを調整する学習法 |
| anon key | Supabase の匿名アクセス用公開 API キー |
| RLS | Row Level Security。PostgreSQL の行単位アクセス制御機能 |
| SPA | Single Page Application。単一の HTML で動作する Web アプリ |
